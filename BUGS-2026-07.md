## Critical — data loss, wrong data, crash on primary path, or exploitable

**C1 — Cache middleware returns wrong data for different nested payloads `[×2]`** · `client/src/client/middleware/cache.ts:87` · certain, **empirical**, **LIVE**
`JSON.stringify(payload, Object.keys(payload).sort())` misuses an *array* as a replacer, which filters keys at **every** depth. `{filter:{a:1}}` and `{filter:{b:2}}` both key as `{"filter":{}}` → the second caller receives the first's cached response. Also throws `TypeError` on null/primitive payloads. MiniMongo enables this middleware by default (5-min TTL). Highest-priority correctness bug.

**C2 — `client.exec` of an explicit `client.chain` crashes** · `client/src/procedures/ref.ts:735-748`, `core/index.ts:139` · certain, **empirical**
`exec()` hydrates `ref.input` without `skipImplicitChain`, hijacking the `steps` array into an inner implicit chain whose output replaces `steps`; the outer handler then throws `steps is not iterable` — *after* the steps already executed via `Promise.all`. The module's own docstring example crashes.

**C3 — HTTP loses the entire request payload by default** · `client/src/adapters/http/client/transport.ts:117` + `shared/strategies.ts:196` · certain
The default strategy maps any procedure ending in `get`/`list`/`find`/`watch` to HTTP GET, the client skips the body for GET, and the payload is never encoded into the query string. Server-side default (`/api/:service/:id?`) also infers the operation from the verb, so `list`→`get` and RPC-named ops → `create` → `HandlerNotFoundError`. Client and server HTTP defaults are mutually incompatible (**H-tier corollary**).

**C4 — `client-git`: shell injection across all 26 procedures** · `client-git/src/procedures/git/*` · certain (security)
Every procedure builds a command string and runs `execSync`, interpolating unescaped user input. Verified vectors: `commit.ts:36` message (`"$(rm -rf ~)"`), `add.ts:20` paths, `checkout.ts:19` ref, `remote.ts:20` name/url, `branch.ts:22` name, plus push/pull/clone/log/diff/init. `client-git` doesn't use `client-shell` at all. Fix: argv execution.

**C5 — `mark lib new` produces packages named `"unnamed"`** · `client-lib/src/procedures/lib/new.ts:156-177` + `cue/npm/package/base.cue:14` · certain, **empirical**
`lib new` runs `cue-config init/generate` but never sets the package name; the CUE default is `*"unnamed"`. Output also lacks `register.ts`, `client.procedures`, `postinstall`, and a test script. **`CLAUDE.md` mandates this exact command for all new packages**, and the `documentation` package is itself named `"unnamed"` as a result.

**C6 — `snapshot.create` crashes before archiving** · `client-snapshot/src/procedures/snapshot/create.ts:63,71,76,233` · certain
`require("fs")` in a `"type":"module"` package; `require` is undefined in ESM scope → `ReferenceError` on every call. Compiled `dist` retains the `require(`.

**C7 — `snapshot.create` tars from the wrong directory** · `create.ts:273-286` · certain
`cwd: workDir` is a fresh temp dir the repos aren't inside, so tar entries don't exist → ENOENT. Doubly dead with C6.

**C8 — collections `ArrayDeque` loses all elements on growth** · `.../impl/array-deque.ts:395,410-411` · certain
`grow()` fires when `_head === _tail` (full), at which point the `size` getter returns 0; `_tail` is set to 0, making copied elements unreachable. Inserting a 9th item into a default deque empties it. *(Dead code — no live consumer, but ships in two packages.)*

**C9 — collections `PriorityQueue` broken from construction** · `.../impl/priority-queue.ts:68,77` · certain
`new Array(capacity)` yields a sparse array; `size` returns `_heap.length` (11), `isEmpty` is false, `peek`/`poll` return `undefined`. Real elements are buried behind holes. *(Dead code.)*

**C10 — collections `LinkedHashMap` corrupts on first collision/resize** · `.../impl/linked-hash-map.ts:225-237,547-559` · certain
One `prev/next` pair is shared between the hash-bucket chain and the insertion-order list; a collision insert loses the older entry, and `resize()` truncates the map to one entry. This is the structure the README advertises for LRU caches. *(Dead code.)*

**C11 — collections `AsyncQueue` drops elements / hangs takers** · `.../async/async-queue.ts:370-376,216-221` · certain
The putter resolver stores no element, so a backpressured producer's item is never enqueued when the putter is fulfilled; the direct-handoff branch resolves the putter without resolving the taker (permanent hang). Sinks the entire CSP `Channel` layer. *(Dead code.)*

**C12 — collections stream collectors always return their seed** · `.../fx/collectors.ts:57-66` · certain
`collect()` discards the accumulator return value, but `counting`/`summingNumber`/`minBy`/`maxBy`/`reducing`/`first`/`last` are value-returning. `collect(xs, counting())` returns 0 for any input. *(Dead code.)*

## High — feature broken, silent failure, resource leak, or build-broken

**H1 — `withContext()` child clients crash on `exec()`/`route()`** · `client/src/client/client.ts:158-171` · certain, **empirical**
`createChild` uses `Object.create` + copies 6 fields; class-field initializers (`procedureRegistry`, `routeResolver`, `batchExecutor`) never run → `TypeError: Cannot read properties of undefined (reading 'get')`.

**H2 — `client.conditional` executes both branches' side effects** · `ref.ts:687-727`, `core/index.ts:255-297` · certain, **empirical**
Eager hydration runs `$immediate` refs before the handler evaluates the condition. Same defect voids `and`/`or` short-circuiting and `tryCatch` (**H3**) and `parallel` error reporting (**M-tier**). See AUDIT §4.2.

**H3 — `tryCatch` core procedure cannot catch** · `core/index.ts:565-585` · certain, **empirical**
The `try` ref is executed during hydration, so a throw propagates before the handler runs; `catch` is never used. Exposed via MCP as `client_tryCatch`.

**H4 — Runtime-defined procedures (`procedure.define`) are uncallable** · `client/src/procedures/define-procedure.ts:204,302` · certain, **empirical**

**H5 — WebSocket client drops all stream frames after the first** · `client/src/adapters/websocket/client/transport.ts:297-368` · certain
`resolve()` fires with a one-item iterable on the first frame; later frames are lost and the pending entry leaks forever. The server side never emits `stream` frames either — WS streaming is broken end-to-end.

**H6 — WebSocket transport reconnects after explicit `close()`** · `.../websocket/client/transport.ts:508-523` · certain
No "intentional close" flag, so `onclose` → `scheduleReconnect()` spawns a new connection after `client.close()`, keeping the process alive.

**H7 — `route({ middlewares })` overrides are dead** · `client/src/client/client.ts:863` + `middleware-override.ts` (243 lines) · certain
The overrides are written to `__middlewareOverrides` but no middleware reads it, and the key shapes mismatch the real middleware. Entirely unused advertised feature.

**H8 — Route output-consumption (`out:`) is dead** · `client/src/client/consumption.ts` + `sponge.ts` (~700 lines) · certain
`RouteResolver.resolve` drops `outputConfig`; `{in, out:{type:"stream"}}` silently behaves as a sponge.

**H9 — Streaming (generator) handlers are unsupported everywhere** · `client.ts:422,902`, `procedure-server.ts:261` · certain
All execution sites `await procedure.handler(...)`; a generator handler returns the generator object, which is then schema-validated/serialized into an error. The `GeneratorHandler`/`streaming`/`outputMode` types advertise a capability with no execution path.

**H10 — Per-attempt timeout poisons subsequent retries** · `client/src/client/middleware/timeout.ts:230-247` · certain
`context.message.signal` is replaced with the composed signal and never restored; after one timed-out attempt the aborted signal becomes the base for every later attempt → instant aborts.

**H11 — Circuit breaker never sees error-status responses** · `client/src/client/middleware/circuit-breaker.ts:243-257` · certain
Only thrown errors are recorded, but transports report failures as `status.type==="error"` items without throwing (the client throws *after* middleware), so `recordSuccess()` runs on failures and the circuit never opens.

**H12 — `BatchExecutor.getStreamingResponse` races two consumers over one generator** · `client/src/client/batch-executor.ts:265-296` · certain
