# Client Ecosystem Audit — July 2026

> A ground-truth audit of the `@mark1russell7` procedure ecosystem: the 48 packages
> listed in `ecosystem/ecosystem.manifest.json`. Read-only analysis; no code changed.
> Companion documents: [BUGS-2026-07.md](./BUGS-2026-07.md) (verified defect register),
> [ROADMAP-2026-07.md](./ROADMAP-2026-07.md) (prioritized remediation plan).

**Audit date:** 2026-07-10 · **Scope:** ~110k LOC across 48 manifest packages · **Method:** eight parallel deep-reads, one per subsystem, each verifying findings against actual source (several confirmed empirically by executing built `dist/`), then cross-validated and spot-checked by hand.

**Explicitly out of scope** (not part of the client system): Vex, tiller, money-machine, MetroGraph, Graph, Graph2, courier, Jqy, meds, lag, notes, render, template, otel-ts, angular-19-playground, figma-diagrams, grafana-infra.

---

## 1. Executive summary

The ecosystem is an ambitious, unusually **consistent-by-construction** monorepo-of-repos: one core RPC/procedure framework (`client`), a config generator (`cue`) that centralizes every package's tsconfig, a CLI (`mark`) that reflects over a global procedure registry, and ~40 satellite packages that register path-addressed, (nominally) Zod-validated procedures wrapping tools and services. The type hygiene is excellent, the layering *intent* is clear, and the MCP integration that exposes procedures to Claude Code as `dev-tools` genuinely works.

But the ecosystem is in a **half-finished, drifted state**, and the gap between what the documentation claims and what the code does is large. The single most important structural fact:

> **The two execution paths in the core diverge, and the "advertised" one is broken in many places.** `call()`/`stream()` (remote, middleware-wrapped) and `exec()`/`route()` (local, direct registry) share almost nothing. Most flagship features — composable control-flow procedures (`chain`/`conditional`/`tryCatch`), runtime-defined procedures, streaming handlers, route middleware overrides, output consumption — are wired into only one path or neither, and several crash on their own documented examples.

Health by the numbers:

- **~150 verified defects**, including **~20 critical** (silent data loss, wrong-data caching, crash-on-primary-workflow, and a shell-injection class).
- **42 of 48 packages have zero tests.** The 344 tests that exist in `client` all target leaf procedure handlers and bypass the composition layer where the critical bugs live.
- **~22 repos have uncommitted working trees**, stranded for ~5 months by a half-landed refactor (see §5.1). ~20 have `dist/` older than `src/` — and `dist/` *is* the published artifact (everything installs via `github:…#main`).
- **~85% of the 15k-LOC collections framework is dead code**, duplicated byte-for-byte in two places.
- The workspace's own `CLAUDE.md` mandates scaffolding packages with `mark lib new` — which **produces packages literally named `"unnamed"`**. The `documentation` package is itself named `"unnamed"`, proving the point.

None of this is catastrophic *for a single-user dev environment* — the live attack surface is a stdio MCP server, not a network port, and the owner knows the terrain. But the ecosystem cannot currently be handed to another person or safely depended upon by an external consumer, and its documentation actively misleads. The roadmap in the companion doc is about converting an ambitious prototype into something trustworthy.

**Overall health grade: C‑.**

## 2. What the system actually is (reality map)

The existing `documentation/ARCHITECTURE.md` describes the intended design. Here is what the code actually implements, corrected.

### 2.1 Layers (as built)

```
Entry points:   mark CLI  ·  impl-mcp-dev (stdio MCP → Claude "dev-tools")  ·  server (HTTP/WS)  ·  direct import
                     │              │                                             │
Bundles:        bundle-dev (shell,fs,cli,pnpm,lib,git,dag,procedure)   bundle-mcp (curated + shell*, see §7)
                     │
Registry:       PROCEDURE_REGISTRY — a globalThis Symbol.for singleton; packages self-register on import
                     │
Wrappers:       client-shell (foundation) · client-git/pnpm/docker/cli/node/vite/vitest/test
                client-fs/s3/mongo/sqlite/snapshot · client-cue · client-lib · client-logger · client-splay
                     │
Core:           client (Client, transports, middleware, ~250 built-in procedures, server, events, components)
Foundations:    cue (config gen) · logger (unused by core) · ecosystem (manifest + dead loader) · mcp · splay
```

### 2.2 Corrections to the documented model

| Documented claim | Reality |
|---|---|
| "47 packages" | **48** in the manifest. `otel-ts` exists on disk but is unlisted. |
| "~100+ procedures" | **~250+.** The `client.*` combinator family alone (array/math/string/object/type/meta/core) is ~108 procedures and is entirely absent from `PROCEDURES.md`. |
| bundle-dev "includes everything: docker, mongo, sqlite, testing" | **False.** bundle-dev = shell, fs, cli, pnpm, lib, git, dag, procedure. No docker/mongo/sqlite/s3/vitest/test/snapshot. |
| "client depends on logger" | **False.** `client` depends only on `ws`. It uses 188 raw `console.*` calls; the `logger` package is wired to nothing. |
| client-dag registers `dag.build`/`dag.visualize` | **client-dag registers zero procedures.** It's a data-structure library. `dag.traverse` is registered by `client-lib`. |
| MiniMongo = in-memory mongo-like store | **A React 19 + Vite visual MongoDB explorer.** No query engine, no store. |
| CLI wrappers "use client-shell to execute commands" | **client-git ignores client-shell entirely** and shells out via `execSync` with string interpolation (see §7). |

### 2.3 The three (four) codegen systems

"Create/refresh a package" is implemented **four** times: (a) `client-lib`'s imperative `lib.new`/`lib.refresh` — the only one wired to `mark`; (b) `client-lib`'s declarative `agg.*` mirror — **unrunnable**, it calls procedures that were never registered; (c) `scaffold`'s Angular schematics — used by nothing, drifted; (d) `cue-config` underneath them all. `scaffold` even hardcodes its own copy of `cue/features.json` and invented a `--inject` CUE convention `cue` never implemented.

## 4. Cross-cutting themes

These recur across many packages and were independently surfaced by multiple audit tracks. They are the highest-leverage things to fix because each one touches the whole fleet.

### 4.1 Two execution paths that don't meet
`call`/`stream` run through middleware + a transport; `exec`/`route` execute registry handlers directly with an empty context. Retry, cache, auth, timeout, tracing, and context/signal propagation apply **only to remote calls**. `procedure.define` writes to a third store (`runtimeProcedures`) that neither path reads. This split is the root cause of a large fraction of the "advertised feature does nothing" bugs.

### 4.2 Eager hydration breaks every control-flow procedure
Procedure-as-data refs default to `$immediate` and are hydrated (executed) via `Promise.all` **before** the handler runs. So `chain` isn't sequential, `conditional` runs both branches' side effects, `and`/`or` don't short-circuit side effects, `tryCatch` can't catch (the throw happens during hydration), and `parallel` can't report per-task errors. The module docstrings document behavior the runtime contradicts.

### 4.3 The stranded `zodAdapter`/`outputSchema` refactor
~5 months ago, local `zodAdapter`/`outputSchema` helpers were deleted from ~22 packages' `register.ts` in favor of importing them from `@mark1russell7/client`. The core change was committed and pushed; the downstream edits were **never rebuilt and never committed**. Result: `client-connection`, `client-server-mongo`, `client-logger`, `client-splay` fail `tsc` today against their installed core, and committed `dist/` folders mask the break. This is the single biggest hygiene item.

### 4.4 `dist/` is the artifact, and it's stale
Every internal dependency is `github:…#main`, so semver `version` fields are dead metadata and **the committed `dist/` folder is what consumers actually run**. ~20 packages have `dist/` older than `src/`; two (mock-client, mock-logger) have deleted `dist/` uncommitted. There is no CI and no `prepare` hook on most packages, so "commit without rebuild" silently ships stale code.

### 4.5 `sideEffects: false` on registration-by-import packages
At least 11 packages declare `"sideEffects": false` while their entire purpose is the side effect of importing `register.js`. Any tree-shaking bundler would eliminate the registration and produce an empty registry. It works today only because the live path is Node ESM, which ignores the flag. `client-mcp` is the one package that does it correctly.

### 4.6 Pervasive "pass-through" validation
The ecosystem's premise is "procedures with Zod schemas," but `client-mongo`, `client-sqlite`, `client-server`, `client-logger`, and much of core use `outputSchema` (an explicit no-op `parse: (d) => d`) for their **inputs**. Malformed input reaches drivers and SQL raw. Whether even the real schemas run depends on the call path (the server validates; direct handler calls don't).

### 4.7 Zero tests where the risk is
`client-shell` (the foundation every wrapper shells through), `client-fs`, `client-git`, and the entire mcp/server tier have no tests. The tests that exist don't exercise the composition layer, which is why a fully-broken package (`client-snapshot`) and born-broken data structures sit in `main` while typechecking clean.

## 5. Architecture flaws

1. **Duplication at every scale.** The collections framework exists byte-identically in two trees (§6). `client-dag` is vendored wholesale inside `client-lib`. There are three server implementations (one real + two wrappers) and four codegen systems. Manifest-loading logic is copy-pasted in 5+ places while the `ecosystem` package that should own it has zero consumers. Command-builder boilerplate is copy-pasted ~20× across CLI wrappers.

2. **Global-singleton registry with implicit, transitive exposure.** Registration-on-import into a `globalThis` registry is elegant for Node ESM but means you cannot tell what a bundle exposes without tracing every member's side-effect imports — which is exactly how `bundle-mcp` leaks `shell.*` (§7). A bundle's tool surface should be computed and asserted, not implied.

3. **Metadata as a grab-bag.** Middleware contexts, route overrides, user metadata, and transport headers all share one `metadata` namespace; the HTTP header converter then leaks any string field as a header and silently drops object fields.

4. **Large dead/vestigial subsystems.** In core alone: `middleware-override.ts` (243 lines, unused), the `consumption.ts`/`sponge.ts` output machinery (~700 lines, unwired), `validation/schemas/` (~370 lines, never imported). As whole packages: `client-connection`, `mock-client`, `mock-logger`, `scaffold`, `ecosystem/src`, and ~85% of collections.

5. **Name-vs-reality smells.** `MiniMongo` is a UI; `docker-sqlite` contains no Docker code; `client-dag` registers no procedures yet lives in the `client-*` namespace reserved for procedure packages; the `documentation` and (effectively) `minimongo` packages don't match their manifest identities.

## 6. The collections story

`client/src/collections/` (15.2k LOC, embedded in core) and the standalone `@mark1russell7/client-collections` (14.8k LOC) are a **byte-identical fork**. Git history shows the standalone package was extracted on 2025-12-26 with intent to be canonical — but the embedded copy was never deleted and `client` was never repointed, so both have been frozen ~6.5 months and every bug exists twice.

- **Only ~15% is used:** `HashMap` + `lruMap`/`ttlMap`/`compose` (via core's cache middleware, live in MiniMongo) and the storage interface. Everything else — trees, PriorityQueue, ArrayDeque, LinkedList, HashSet, LinkedHashMap, async queue/channels, stream collectors, the behavior proxies — has zero consumers.
- **The unused majority is where the born-broken structures live:** PriorityQueue reports `size 11` on construction; LinkedHashMap shares one pointer between its bucket chain and order list and corrupts on the first collision/resize; ArrayDeque loses all elements when it grows; AsyncQueue drops elements; 8 stream collectors always return their seed value. These have plainly never been executed.
- **It inflated the core's API and caused a self-admitted export conflict** (`client/src/index.ts` re-exports ~200 collections symbols from the RPC client root, with a comment noting middleware name clashes).
- **A split-brain type seam:** `client-mongo` implements `CollectionStorage` from the *package* while core consumes the *embedded copy's* — safe today only because the files are identical and TS is structural.

Recommendation: finish the extraction, delete the embedded copy, and shrink the surviving package to its load-bearing ~15%. Details in the roadmap.

## 7. Security posture

This is a single-user dev machine, and the **live** surface — the stdio `impl-mcp-dev` server Claude Code talks to — is not networked. The findings below are calibrated to that reality (blast-radius notes, not alarms), but none of the mitigations are enforced by code.

1. **`bundle-mcp` exposes arbitrary command execution to the model.** Its header claims it excludes `shell.*` "since Claude already has shell access," and it omits `client-shell` from its dependencies — but `client-cli`, `client-docker`, and `client-test` each `import "@mark1russell7/client-shell"` as a side effect, so `shell.run`/`shell.exec`/`shell.which` register anyway. **Confirmed live:** the running `dev-tools` server exposes `mcp__dev-tools__shell_run`. The bundle's core design premise is silently violated.

2. **The networked `server` binds `0.0.0.0` with no auth and wildcard CORS**, serving the entire registry — including `shell.run` and `fs.write` — to anyone on the LAN. `server.create` defaults to `{host: "0.0.0.0", port: 3000, autoRegister: true}` with no authentication anywhere in the path. The same pattern appears in MiniMongo's dev bootstrap (`0.0.0.0:5000/5001`). Independently flagged by three audit tracks. Mitigation: default `127.0.0.1`, gate public binding behind an explicit flag + bearer token.

3. **A shell-injection class in `client-git`.** All 26 git procedures build commands via `execSync` with string interpolation of user input (`git commit -m "…"`, unquoted refs/paths/branches/URLs). Concrete break: `client.call(["git","commit"], {message: "$(rm -rf ~)"})` executes the substitution. `client-pnpm`, `client-docker`, and `client-test` have the same class via `shell.exec` + `join(" ")`. The fix (route through `shell.run` with argv arrays, as `client-cli` already does) eliminates the entire class.

4. **Command injection via snapshot metadata** (`snapshot.diff` runs `git diff --stat ${commit}..HEAD` where `commit` comes from JSON downloaded from S3) and **RCE-by-design** in core's `createDynamicHandlerLoader` (dynamic-imports a module path from stored procedure records, no allowlist). Low exploitability today; worth an allowlist before any multi-user use.

5. **Infra with no auth:** docker-mongo (MongoDB 8, no root credentials, published port) and docker-sqlite's `sqlite-web` UI (:8080, full read/write). Fine behind localhost; bind published ports to `127.0.0.1`.
