# Audit: `@mark1russell7/client` core package (excluding src/collections)

## Package summary

**Purpose**: Foundational package of a personal "procedure ecosystem". Provides: a protocol-agnostic RPC `Client` with onion-model middleware; a global `PROCEDURE_REGISTRY` of path-addressed, Zod-validated procedures; procedure-as-data (`$proc`/`$ref`/`$when` JSON refs + hydration); transports (HTTP/WebSocket/Local/Mock, client+server sides); a `Server`/`ProcedureServer`; an event bus; a component-as-procedure system; procedure persistence/sync; and a CLI (`discover`/`announce`/`init`).

**Module map** (in-scope ~24.5k LOC):
- `src/index.ts` — root barrel; re-exports collections + everything else (~370 lines of exports).
- `src/client/client.ts` (966) — `Client`: `call/stream/exec/route/routeStream`, `use/withContext/schema/useRegistry`; per-call middleware composition.
- `src/client/types.ts`, `context.ts`, `call-types.ts`, `consumption.ts`, `sponge.ts`, `route-resolver.ts`, `batch-executor.ts`, `middleware-override.ts`, `typed.ts`, `typed-method.ts` — message/status types, context merging, nested-route batch API, output-consumption machinery (largely unwired).
- `src/client/middleware/*` (10 files) — retry, cache, timeout, auth, tracing, circuit-breaker, rate-limit, batching, pagination, + context type declarations.
- `src/client/validation/*` — Zod-like schema middleware + registry; `validation/schemas/` (message/transport) is dead code (never imported).
- `src/client/errors/*` (~1.6k) — error-code registry with rich metadata; factories; HTTP mappings. Solid, data-heavy.
- `src/middleware/*` — generic sync/async middleware compose/bundle with type-level context tracking. Solid.
- `src/procedures/*` — types, `define.ts` builders, `registry.ts` (globalThis-symbol singleton), `ref.ts` (procedure-as-data + hydration), `define-procedure.ts` (runtime meta-procedures), `core/*` (chain/parallel/conditional/logic/math/string/object/array/meta), `storage/*` (serialize/sync registry), `collection/*` (CRUD procedure factories), `cli/*`.
- `src/adapters/*` — http client+server (Express), websocket client+server (`ws`), local, mock.
- `src/server/*` — `Server` (matcher-based handler registry + middleware), `ProcedureServer` (registry→handler bridge, RepositoryProvider).
- `src/events/*` — `DefaultEventBus` (pub/sub, buffering, `stream()` async iteration). Solid except minor races.
- `src/components/*` — components compiled to procedures under `["components", ...]`.

**Dependency on src/collections**: `procedures/types.ts` (`CollectionStorage` in `RepositoryProvider`), `client/middleware/cache.ts` (`compose/lruMap/ttlMap/hashMap`), `procedures/storage/factory.ts` (`InMemoryStorage/ApiStorage/HybridStorage`), `server/procedure-server.ts` (`CollectionStorage`), and `src/index.ts` (`export * from "./collections/index.js"`). Cache middleware is the only *behavioral* coupling in the client core.

**Build/test state**: `tsc` strict (nodenext, exactOptionalPropertyTypes, etc.) passes; dist is current. `vitest run`: 344 tests / 10 files, all passing — but **every test file lives in `src/procedures/`** (core procedure handlers + ref + define-procedure). Zero tests for `Client`, transports, middleware, server, events, components. Tests call handlers directly, bypassing exec/hydration, so the composition-layer bugs below go undetected.

**Health grade: C-.** Excellent type hygiene, consistent style, strong error-metadata system, working unit tests for leaf procedures — but multiple flagship, documented flows (exec+chain, conditional, route overrides, HTTP default framing, WS streaming, runtime-defined procedures, `withContext` children) are broken as verified below.

## Bugs

Empirically verified = run against `dist` on Node 25.

1. **`withContext()` child clients crash on `exec()`/`route()`** — `src/client/client.ts:158-171` + 646. `createChild` uses `Object.create(Client.prototype)` and copies only 6 fields; class-field initializers (`procedureRegistry = PROCEDURE_REGISTRY`, `routeResolver`, `batchExecutor`) never run (`useDefineForClassFields: true`). `execInternal` → `this.procedureRegistry.get(path)` → `TypeError: Cannot read properties of undefined (reading 'get')`. **Verified empirically.** Severity: **high**. Confidence: certain.

2. **`client.exec` of an explicit `client.chain` ref crashes** — `src/procedures/ref.ts:735-748` (implicit-chain detection) + `src/procedures/core/index.ts:139`. `exec()` hydrates `ref.input` without `skipImplicitChain`, so `{steps: [refA, refB]}`'s array is hijacked into an *inner* implicit chain; its `ChainOutput` object replaces `steps`; the outer chain handler then throws `TypeError: steps is not iterable` — after the steps have already executed (in parallel, via `Promise.all` at ref.ts:750). The module's own docstring example (core/index.ts:22-29) crashes. **Verified empirically.** Severity: **critical** (flagship composition API). Confidence: certain.

3. **`client.conditional` executes both branches eagerly** — `src/procedures/ref.ts:687-727` + `src/procedures/core/index.ts:255-297`. Hydration runs `$immediate` (default) refs before the conditional handler sees them; with `condition:false`, the `then` branch's side effects still ran (`calls: ["sideThen","sideElse"]`). Same applies to `and`/`or` "short-circuit" (`core/logic.ts:121-127`) — short-circuiting is value-only; all side effects happen. **Verified empirically.** Severity: **high**. Confidence: certain.

4. **Named-context deferral (`$when: "name"`) has inverted semantics when nested** — `src/procedures/ref.ts:380-405` (`return contextStack.includes(when)`) + 717 (name pushed *before* hydrating the named ref's input). A ref with `$when:"traversal"` nested under a `$name:"traversal"` ref is executed *during hydration* (verified: `visit` arrived as `"a-result"`, not as data), the opposite of the module doc (ref.ts:29-46: "The inner $proc is NOT executed during hydration"). Top-level via `exec()` accidentally defers only because `normalizeRef`/`fromJson` (ref.ts:546-554) drop `$name`. **Verified empirically both ways.** Severity: **high**. Confidence: certain.

5. **`fromJson`/`toJson`/`extractTemplate`/`parseProcedureJson` silently drop `$when` and `$name`** — `src/procedures/ref.ts:546-563, 836-873, 893-909`. Round-tripping a composition through JSON loses all execution-timing semantics (deferred refs become `$immediate`). Severity: **high** (silent behavior change). Confidence: certain (code); root cause of #4's asymmetry.

6. **Runtime-defined procedures (`procedure.define`) are uncallable** — `src/procedures/define-procedure.ts:204, 302` stores in a private module-level `runtimeProcedures` Map that neither `Client.execInternal` (client.ts:397 reads only `this.procedureRegistry`) nor any transport/server consults. The file's own doc example (`client.call(["my","workflow"], ...)` after define) fails `No handler registered for :my.workflow`. **Verified empirically.** Severity: **high**. Confidence: certain.

7. **`ProcedureServer.registerCollectionProcedures` always throws** — `src/server/procedure-server.ts:344`: `const { createCollectionProcedures } = require("../procedures/collection/procedures");` in an ESM package (`"type":"module"`, module nodenext). **Verified empirically** (`Cannot find module`). Severity: **high**. Confidence: certain.

8. **Cache middleware: wrong-result cache collisions + crash** — `src/client/middleware/cache.ts:87`: `JSON.stringify(payload, Object.keys(payload as object).sort())` uses an *array replacer*, which filters keys at **every** depth. Different nested payloads collide: `{filter:{a:1}}` and `{filter:{b:2}}` both key as `{"filter":{}}` → cache returns the wrong response. Null/primitive payloads throw `TypeError` (`Object.keys(null)`). **Verified empirically.** Severity: **critical** (silent wrong data). Confidence: certain.

9. **WebSocket client transport drops all stream frames after the first** — `src/adapters/websocket/client/transport.ts:297-368, 400-424`. `send()` awaits one promise; `handleMessage` calls `pending.resolve(iterable-of-ONE-item)` on the first frame; later frames re-`resolve` (no-op) and are lost; for `stream.done=false` frames the pending entry also leaks in `pendingRequests` forever. Streaming over WS is broken end-to-end (server side never sends `stream` frames either — `server/transport.ts:299-317` sends exactly one response). Severity: **high**. Confidence: certain (code).

10. **WebSocket transport reconnects after explicit `close()`** — `src/adapters/websocket/client/transport.ts:508-523` sets state/`ws=null` but the socket's async `onclose` → `handleClose` (161-185) still runs `scheduleReconnect()` (no "intentional close" flag), spawning a new connection after `client.close()`. Severity: **high** (resource leak, keeps process alive). Confidence: certain (code).

11. **Per-request timeout mislabeled as `connectionTimeout`** — `transport.ts:402-406`: every RPC gets a 10s response deadline from `connectionTimeout`; any slower call rejects "Request timeout" even on a healthy connection. Severity: medium. Confidence: certain.

12. **`route({ middlewares })` overrides are dead** — `src/client/client.ts:863` writes `__middlewareOverrides`, but no middleware reads it (grep: only `middleware-override.ts` itself). Key shapes also mismatch the real middleware: `RetryOverride.attempts` vs retry's `metadata.retry.maxAttempts` (retry.ts:106), `TimeoutOverride.ms` vs `timeout.overall`. Additionally, procedures with local handlers bypass the middleware chain entirely (client.ts:887-922), so overrides could never apply there anyway. The entire `middleware-override.ts` module (243 lines) is exported but unused. Severity: **high** (advertised feature is a no-op). Confidence: certain.

13. **Route output-consumption config (`out:`) is dead** — `src/client/consumption.ts` + `src/client/sponge.ts` (~700 lines): `flattenRoute` extracts `outputConfig` (call-types.ts:272-313) but `RouteResolver.resolve` drops it (route-resolver.ts:134-138, `ResolvedRoute` has no such field) and `BatchExecutor`/`executeProcedure` never consume it. `consumeGenerator`/`spongeWithHandlers` are referenced nowhere outside sponge.ts. `{ in, out: {type:"stream"} }` silently behaves as sponge. Severity: **high** (dead feature). Confidence: certain.

14. **Generator (streaming) procedure handlers are unsupported everywhere that executes procedures** — `client.ts:422` (`execInternal`), `client.ts:902` (`executeProcedure`), `procedure-server.ts:261` all do `await procedure.handler(...)`; for a generator handler this returns the generator object itself, which is then schema-validated/serialized (→ `OUTPUT_VALIDATION_ERROR` or `{}` over the wire). The `GeneratorHandler` type, `Procedure.streaming`, `outputMode` matrix (procedures/types.ts:283-311) and the sponge machinery advertise a capability with no execution path. Severity: **high**. Confidence: certain (code).

15. **HTTP default configuration silently loses request payloads** — `src/adapters/http/client/transport.ts:117` skips the body for GET; default `restfulHttpMethodStrategy` (shared/strategies.ts:196-221) maps `get/list/find/watch` → GET; `defaultUrlPattern.format` never encodes the payload into query params; the server side (http/server/transport.ts:212-221) only reconstructs `{id}` from `req.params["id"]`, which is never set by the regex catch-all route. Any procedure whose last path segment is get/list/find/watch loses its entire input over HTTP by default. Severity: **critical** for HTTP users. Confidence: certain (code, both sides read).

16. **Client and server HTTP defaults are mutually incompatible** — client default `defaultUrlPattern` emits `/{service-with-dots}/{operation}` (e.g. `/collections.users/list`), while server default `defaultServerUrlStrategy` (http/server/strategies.ts:20-55) hardcodes `/api/:service/:id?` and *infers the operation from the HTTP verb* (GET+id→"get", POST→"create"). `list` → parsed as `get`; any RPC-named op (e.g. `scan`) → `create` → `HandlerNotFoundError`. The true inverse (`defaultUrlPattern.parse`) exists but is unused by the server. Severity: **high**. Confidence: certain.

17. **HTTP response parse fallback is impossible** — `http/client/transport.ts:210-217`: on `response.json()` failure it calls `response.text()`, but the body was already consumed → `TypeError: Body is unusable` replaces the real error. Severity: medium. Confidence: certain.

18. **All error responses become HTTP 500** — `http/server/transport.ts:241-252`: `status.code` for errors is a string code (`"NOT_FOUND"`, `"VALIDATION_ERROR"`); `parseInt` → NaN → 500. The rich `ERROR_REGISTRY` `httpStatus` metadata is never consulted. (Only `HandlerNotFoundError` maps to "404" via server.ts:323-334.) Severity: medium. Confidence: certain.

19. **`BatchExecutor.getStreamingResponse` races two consumers over one generator** — `src/client/batch-executor.ts:265-296`: `results: wrappedStream()` and the `complete` promise each iterate wrappers around the *same* `stream` generator; items are distributed round-robin between whichever consumer polls first, so a caller iterating `results` observes only a subset (complete's response stays whole only because both wrappers push into the shared array). Severity: **high**. Confidence: certain (code).

20. **`client.call()` abandons the response stream** — `src/client/client.ts:297-307`: takes the first item and never calls `iterator.return()`; remaining items are silently dropped and transport-side generators/finally blocks never complete (leak for streaming transports). **Verified empirically** (second item dropped). Severity: medium. Confidence: certain.

21. **Per-attempt timeout poisons subsequent retries** — `src/client/middleware/timeout.ts:230-247`: `context.message.signal` is replaced with the composed signal and never restored in `finally`. With retry (outer) + per-attempt timeout (inner), after one timed-out attempt the *aborted* composed signal becomes the base signal for every later attempt → instant aborts. Also `retry.ts:122` then throws "Request was aborted". Severity: **high**. Confidence: certain (code).

22. **Retry middleware buffers entire streams** — `retry.ts:130-135` collects all items before yielding any; defeats streaming, unbounded memory, and re-executes the whole stream on retry. Also only `maxAttempts` is context-overridable (retry.ts:106-109) though docs claim contextual override generally. Severity: medium. Confidence: certain.

23. **Circuit breaker never sees error-status responses** — `circuit-breaker.ts:243-257`: only *thrown* errors are recorded; transports (HTTP/local/WS) report failures as `status.type==="error"` items without throwing (client throws *after* middleware, at client.ts:517), so `recordSuccess()` runs on failures and the circuit never opens in the standard stack. Also HALF_OPEN reopen requires `failures.length >= failureThreshold` (line 165) instead of one failure. Severity: **high** (feature ineffective). Confidence: certain (code).

24. **Adaptive batching creates a new queue per request** — `batching.ts:338-356`: `createBatchingMiddleware(...)` is invoked inside the `(next)=>` closure, and `Client.composeMiddleware` (client.ts:509, 531-541) re-applies middleware on **every** `stream()` call → fresh `BatchQueue` per request → nothing ever batches; each request just waits `maxWaitTime`. `currentBatchSize` adjustments never propagate. (Base `createBatchingMiddleware` also never actually batches — `processBatch` at 227-246 admits it executes requests individually.) Severity: medium (feature no-op + added latency). Confidence: certain.

25. **`sponge` `arrayAccumulator` duplicates every element** — `sponge.ts:43-46` pushes `current` into `accumulated`, but `sponge()` (104-107) already pushed it into `values` before calling the accumulator → `[v1,v1,v2,v2,...]`. `mergeAccumulator` similarly double-merges. Severity: medium (if this machinery is ever wired up). Confidence: certain.

26. **Generic collection procedures are unreachable** — `procedures/collection/procedures.ts:258, 277, 296` register literal paths `["collections","*","get"]` etc.; `ProcedureRegistry.get` is exact-match (registry.ts:172-174) with no wildcard resolution (`matchPath` exists only in route-resolver and is unused here). Only a literal call to `collections.*.get` would match. Severity: medium (dead feature). Confidence: certain.

27. **Child-component rendering ignores namespaces** — `components/define.ts:368`: `PROCEDURE_REGISTRY.get(["components", type])` — namespaced components (`["components", ns, type]`) can never be rendered as children; streaming child generators also aren't closed after first yield (387-393). Severity: medium. Confidence: certain (code).

28. **`registerFromRegistry` throws on stub procedures, contrary to its own contract** — `procedure-server.ts:185-187` throws `"has no handler"`; `registerFromRegistry` (328-334) says "Only registers procedures that have handlers" but calls `registerProcedure` unconditionally — one stub in the global registry (e.g. synced-from-storage stubs, which always lack handlers per serialization.ts:126-155) aborts server startup. 32. **Route resolver discards parsed input** — `route-resolver.ts:117-138`: validates with `safeParse` but pushes `input: entry.leaf` (raw), so Zod defaults/coercions/transforms never reach handlers via `route()`; inconsistent with `execInternal` which uses `inputResult.data` (client.ts:422). Also for `{ in, out }` leaves, validation runs against `entry.input` but execution receives `entry.leaf` (the whole `{in,out}` wrapper) as handler input — handlers get `{in:..., out:...}` instead of the input. Severity: **high** (route() passes wrong input shape for new-style leaves). Confidence: certain (code: resolve() line 104 destructures `input` for validation but line 137 stores `entry.leaf`).

## Architecture flaws

- **Two disconnected execution paths.** `call/stream` (middleware + transport) vs `exec/route` (direct registry handler execution, `client.ts:393-436, 879-949`) — the latter bypasses middleware, metadata, signals, and context entirely (`execInternal` builds `ctx = {metadata: {}}`). Retry/cache/auth/timeout apply only to remote calls; per-call `CallOptions` don't exist for `exec()`.
- **Eager hydration vs control-flow procedures is a fundamental mismatch.** chain/conditional/and/or/tryCatch/parallel all assume deferred refs, but the default `$when` is `$immediate` and hydration runs arrays via `Promise.all` — sequential semantics, short-circuiting, and error scoping are all illusions unless every step is annotated `$when:"$parent"` (which no doc example does). Bugs 2-5, 34-35 are symptoms of this one design fault.
- **Dead/vestigial subsystems**: `middleware-override.ts` (243 lines, unused); `consumption.ts`+`sponge.ts` output-config machinery (~700 lines, unwired); `validation/schemas/` (~370 lines, never imported); `core/results.ts` StepResult/ChainResult types (unused by the actual chain); `getBatchingStats`/`getCircuitBreakerStats` return `null` placeholders; `SyncedProcedureRegistry.connect()` sets flags but connects nothing; generic `collections.*` procedures unreachable; `batching` middleware executes serially by admission.
- **Three separate procedure stores**: `PROCEDURE_REGISTRY`, `runtimeProcedures` (define-procedure.ts), and `LocalTransport`'s handler map — with no bridging. `ProcedureServer` also doesn't subscribe to registry `register` events, so late registrations never become server handlers.
- **`Client.use()` mutates shared state while claiming immutability** — `use()` pushes onto `this.middleware` and returns `this` cast to a new type; children share the parent's array by reference, so `child.use(mw)` retroactively mutates the parent. Type accumulation is fictional at runtime.
- **`composeMiddleware` per call** (client.ts:509) re-invokes every middleware factory per request — any middleware holding state in the `(next)=>` closure resets per call (bit adaptive-batching; a footgun for user middleware).
- **Duplicated helpers**: `pathToMethod` duplicated in client.ts:954 and procedure-server.ts:361; `anySchema` re-declared in schemas.ts, serialization.ts, array.ts, meta.ts (and more); mock-client test helper copy-pasted across 5+ test files; `resolveInputRefs` in define-procedure.ts re-implements ref.ts hydration with different (subset) semantics, including a `$last` branch that can never receive data.
- **Error-handling gaps**: procedures/index.ts:205-225 swallows all auto-registration errors; local transport marks every handler error `retryable:false` with only `message` preserved (stack/code lost); `Server.errorToResponse` mixes numeric-string codes ("404"/"500") with symbolic codes, and the rich `ERROR_REGISTRY` metadata is never attached to any transport error path (only HTTP client uses `createHTTPStatusError`).
- **`Metadata` as a grab-bag**: middleware contexts, overrides (`__middlewareOverrides`, `__schema`, `__validation`), user metadata, and transport headers all share one namespace; `DefaultHeaderConverter` then leaks any string field as an HTTP header while silently dropping object fields.
- **Registry `pathToKey` ambiguity**: `["a.b","c"]` collides with `["a","b","c"]`; `validateProcedure` forbids dots but `registry.register` never calls it.
- **`ws` is the only runtime dependency but the WS *client* uses the global `WebSocket`** (Node ≥21 only, engines say ≥20), while the server uses `ws` — inconsistent and breaks on Node 20.

## Doc drift

- **README API reference is systematically stale** (README.md:313-417): `RetryOptions {maxAttempts, initialDelay, backoffFactor, jitter:boolean, retryableErrors}` vs actual `{maxRetries, retryDelay, jitter:number, shouldRetry, onBeforeRetry, onAfterRetry}`; `CacheOptions {maxSize, keyFn, storage}` vs `{capacity, keyGenerator, onStats}`; `createOverallTimeoutMiddleware(timeout:number)` vs options object; `createApiKeyAuthMiddleware(apiKey, headerName?)` vs single arg; `CircuitBreakerOptions {threshold, timeout}` vs `{failureThreshold, failureWindow}`; `RateLimitOptions {maxTokens, refillRate}` vs `{maxRequests, window, strategy}`; Zod middleware `{mode:"input"|"output"|"both", throwOnError}` vs `{mode:"strict"|"warn"|"silent", transformOutput}`.
- README transport options wrong: `HttpTransportOptions.headers/fetch` (actual `defaultHeaders`, no fetch injection); flat `WebSocketTransportOptions.reconnectDelay/heartbeatInterval` (actual nested objects); `LocalTransportOptions.handlers: Map` (actual `Record`); `HttpServerTransportOptions {port, hostname}` omits the *required* `app`; `WebSocketServerTransportOptions {port, auth}` vs actual `{server, authenticate}`.
- README:447-453 `PROCEDURE_REGISTRY.delete/list` don't exist (`unregister`/`getAll`); README:687-689 `ERROR_REGISTRY.get/has/list` — it's a plain object; README:458-462 `.when("immediate")` — real constants are `"$immediate"/"$never"/"$parent"`; README:603/762 `bus.stream("user:*")` — no wildcard support exists; README:304-309 mockBuilder `.method().response().times().delay().build()` — actual API is `when/thenReturn/thenError/thenStream` and requires a transport argument.
- `src/index.ts:364-372` and README:692-705 advertise `client/middleware` deep import — blocked by exports map (bug 30).
- `ref.ts:29-46` named-context example documents the opposite of runtime behavior (bug 4); `core/index.ts:1-46` chain/conditional examples crash or double-execute (bugs 2-3); `define-procedure.ts:7-25` "Now use the defined procedure" fails (bug 6); `$last` documented in define-procedure refs but never populated.
- `tracing.ts:49-57` and README architecture examples pass `middleware: [...]` to `new Client({...})` — `ClientOptions` has no such field.
- `typed.ts:13` example `createTypedCaller(client.call.bind(client))` — `Client.call` takes a `Method` object, not a `string[]` path; the example doesn't typecheck or run.
- WS server transport header claims "Supports bidirectional streaming" (websocket/server/transport.ts:5) — it sends exactly one response per request; the client's `stream.done` protocol has no producer.
- `registerFromRegistry` doc-comment contradiction (bug 28); `batching` middleware documents "Combines multiple requests into a single batch" while `processBatch` executes them individually.

## Roadmap candidates (prioritized)

1. **Fix the correctness landmines first**: `createChild` field initialization (bug 1 — one-line fix: initialize registry/resolver lazily via prototype getter or copy in createChild); cache key generator (`JSON.stringify(payload)` with stable stringify, guard non-objects) (bug 8); HTTP GET payload loss — make `postOnlyStrategy` the default or encode query params (bugs 15-16); `require`→`await import` in `registerCollectionProcedures` (bug 7); skip handler-less procedures in `registerFromRegistry` (bug 28).
2. **Decide hydration semantics and make control-flow procedures real**: default chain/conditional/tryCatch/parallel inputs to deferred (`$when:"$parent"` implicitly for known control-flow paths, or execute refs lazily inside handlers).
