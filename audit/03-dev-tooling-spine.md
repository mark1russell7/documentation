All source files in scope were read; findings below were verified against surrounding code, and several were verified empirically (`cue eval`, manifest/disk cross-check script, vitest run, tsc typecheck). No files were modified. Note: `client-lib/dist` and `client-cue/dist` are stale relative to uncommitted `src/register.ts` edits (both repos have WIP switching from a local `zodAdapter` copy to the one exported by `@mark1russell7/client`); line numbers below refer to current `src`.

## Bugs

1. **`lib new` creates a package.json named `"unnamed"`** — `C:\Users\markt\git\client-lib\src\procedures\lib\new.ts:156-177` runs `npx cue-config init` + `npx cue-config generate` in the new dir and never writes the package name; `C:\Users\markt\git\cue\npm\package\base.cue:14` defaults `name: #PackageName | *"unnamed"`; `cue/src/cli.ts:168-170` only keeps an existing name (none exists for a fresh package). Verified by running `cue eval base.cue ts.cue node.cue cue.cue -e output` → `"name": "unnamed"`. Every scaffolded package needs manual rename; also missing `description`, `client.procedures`, `postinstall`, test script vs actual conventions. Severity: **critical** (primary documented workflow produces broken output), confidence: certain.

2. **All top-level aggregation procedures reference procedures that don't exist** — `refresh-aggregation.ts:242,267,282` calls `["lib","refresh.single"]`; `install-aggregation.ts:197,241` calls `["lib","install.cloneMissing"]` / `["lib","install.single"]`; `pull-aggregation.ts:106` calls `["lib","pull.single"]`; `new-aggregation.ts:174` calls `["lib","manifest.add"]`; all four also call `["dag","execute"]`. None are registered anywhere (grep across all `src/`: only mocks in tests match; the real registrations in `register-aggregations.ts:68-216` use *different* paths like `["agg","lib","refresh","single"]`, and only `dag.traverse` exists). The imperative implementations carry `@deprecated … will be removed in v2.0` notices pointing at these broken replacements. Severity: **high**, confidence: certain (missing registration) / likely (runtime failure mode).

3. **`pnpm.onlyBuiltDependencies` auto-population in CUE never emits** — `C:\Users\markt\git\cue\npm\package\base.cue:63-77`: `_allDeps` iterates `output.dependencies`, which is absent in generated output (and `output` is defined in terms of `_allDeps`), so the guarded `pnpm:` field is silently dropped. Verified: `cue eval … -e output` and even `output & {dependencies: {"@mark1russell7/client":"x"}}` emit no `pnpm` key. All the hand-maintained `pnpm.onlyBuiltDependencies` blocks across packages exist because this feature is dead. Severity: high, confidence: certain (empirically verified).

4. **`mark <cmd> -f` is silently discarded and conflicts with procedure shorts** — `C:\Users\markt\git\cli\src\cli.ts:376-384`: `formatOverride` reads only `options["format"]`, then `delete options["f"]` unconditionally before `parseFromSchema`. So `-f` never works as `--format`, and procedures whose `shorts` map `f` (`lib.refresh` force, `lib.audit` fix — `client-lib/src/register.ts:83,143`; `cue.init` force — `client-cue/src/register.ts:44`) silently lose the flag in local mode. In client mode (`client-mode.ts:63-66` runs before the deletion) `-f` *does* map to `force` — behavior differs depending on whether a server happens to be running. Severity: high, confidence: certain.

5. **client-cue can't find the cue package outside its own checkout** — `C:\Users\markt\git\client-cue\src\shared.ts:15-16`: `packageRoot = resolve(__dirname, "..", "node_modules", "@mark1russell7", "cue")`, and `@mark1russell7/cue` is only a **devDependency** (`client-cue/package.json`). Works in the dev checkout (pnpm symlinks direct deps there) but breaks when client-cue is consumed as a dependency (pnpm virtual-store layout has no nested `node_modules`, and the devDep isn't installed at all — e.g. via `bundle-mcp`, which depends on client-cue). All `cue.*` procedures then fail with "Could not load features.json". Fix is `createRequire(import.meta.url).resolve("@mark1russell7/cue/package.json")` + promoting to a real dependency. Severity: high (medium in current source-checkout usage), confidence: likely.

6. **`cue.generate` reports success when CUE evaluation fails** — `C:\Users\markt\git\client-cue\src\procedures\cue\generate.ts:60-64` and `120-124` return `null` on `cue eval` failure (stderr discarded), and `cueGenerate` (lines 196-231) just skips the file and returns `{ success: true, generated: [...] }`. A broken CUE install/template yields a silent partial generate. Severity: medium, confidence: certain.

7. **`cue-config generate` clobbers custom `exports` maps** — `C:\Users\markt\git\cue\src\cli.ts:181-184` (same logic in `client-cue/.../generate.ts:82-85`): any key not in {name, version, description, devDependencies, peerDependencies, scripts, files} is overwritten by the generated value. Generated `exports` is always the single `"."` entry (`npm/package/ts.cue`), so re-running generate in `client` (which exports `./client`, `./components`, `./events`, `./procedures`) or `bundle-dev` (exports `./register.js`) silently breaks consumers. Severity: high (latent, but generate is the documented routine operation), confidence: certain (logic verified; not executed).

8. **DAG `requiredBranch` is a tautology; branch enforcement is dead** — `C:\Users\markt\git\client-lib\src\dag\builder.ts:25-27`: `requiredBranch = info.currentBranch ?? "main"` and `gitRef = info.gitRemote ?? …` (a remote *URL*, not a `github:…#branch` ref, contradicting `types.ts:109-116`). `ensureBranch` (`git/operations.ts:186-215`) therefore always compares the current branch to itself and no-ops. The intended behavior (checkout the branch pinned in dependents' package.json refs — `ref-parser.ts` exists for exactly this and is unused in the DAG path) never happens. Severity: medium, confidence: certain.

9. **`lib.refresh --dry-run` (recursive/all) still executes `ensureBranch`** — `refresh.ts:342-344` and `415-417` call `ensureBranch` *before* `refreshSinglePackage` regardless of `dryRun`; `ensureBranch` auto-commits WIP ("WIP: Auto-commit all changes before branch switch") and checks out branches. Currently masked by bug 8 (it never switches), but the moment 8 is fixed, dry-run becomes destructive. Severity: medium (latent), confidence: certain.

10. **`lib.pull` output always reports `commits: 0`** — `C:\Users\markt\git\client-lib\src\procedures\lib\pull.ts:117-143`: the processor discards `pullSinglePackage`'s result; final results are rebuilt from the DAG result with `commits: 0, // Will be filled from actual pull` — never filled; `fastForward` also lost. Same pattern loses `plannedOperations` for `lib.refresh --all --dry-run` (`refresh.ts:355-367`). Severity: low, confidence: certain.

11. **`lib.rename` depends on `$HOME`, broken on Windows shells** — `C:\Users\markt\git\client-lib\src\procedures\lib\rename.ts:257,263`: `rootPath = process.env["HOME"] + "/git"` → `"undefined/git"` when HOME is unset (cmd/PowerShell); every other procedure uses `os.homedir()`. Procedure exits with "Root path does not exist". Severity: medium, confidence: likely (environment-dependent).

12. **`ecosystem.procedures` imports register files without `file://` on Windows and misattributes namespaces** — `C:\Users\markt\git\client-lib\src\procedures\ecosystem\procedures.ts:178-184`: `await import(join(packagePath, …))` with a raw `C:\…` path throws `ERR_UNSUPPORTED_ESM_URL_SCHEME` (swallowed by the empty catch), unlike `cli/src/ecosystem.ts:133` which converts correctly. ## Architecture flaws

- **Three (really four) parallel codegen systems for the same job.** "Create/refresh a package" exists as: (a) client-lib imperative `lib.new`/`lib.refresh` (the only one actually wired to `mark`), (b) client-lib's aggregation mirror (registered under `agg.*`, unrunnable — bug 2), (c) scaffold's `lib-new`/`lib-refresh` schematics (unused by anything, drifted), and (d) cue-config init/generate underneath them all. scaffold duplicates the feature graph (`src/features/index.ts` hardcodes a copy of `cue/features.json`) and invented a `--inject`-based CUE convention the cue package never implemented.
- **client-lib vendors client-dag.** `client-lib/src/dag/{builder,executor,traversal}.ts` is a near-verbatim (de-genericized) copy of `client-dag/src/dag/*` (diff-verified); `@mark1russell7/client-dag` is declared in client-lib's dependencies and never imported. client-dag also registers no procedures (no register.ts), yet `bundle-dev/src/register.ts:19` imports it "to register procedures" — a no-op import.
- **ecosystem/src is a dead API; manifest logic is copy-pasted 5+ times.** `cli/src/ecosystem.ts:80`, `client-lib` scan/install/audit/new, and `cue/src/cli.ts:523-542` each hardcode `~/git/ecosystem/ecosystem.manifest.json` + `~` expansion; nobody imports `@mark1russell7/ecosystem`.
- **cue logic duplicated between cue and client-cue** (`resolveFeatures`, `determineTsconfig`, `TSCONFIG_PRIORITY`, the package.json merge) — already drifted (`setupCueMod`, error handling, warning on unknown features).
- **Aggregation rollout is half-finished and its test harness hides that.** Deprecation notices on all imperative lib procedures point to the agg versions; the agg versions can't run; parity/aggregation tests (all 154 pass) execute a *reimplemented interpreter inside the test files* against a mock client that returns `{}` for any path — so missing procedures, `~` paths, and `{{template}}` semantics are never exercised against the real `procedure.define`/hydration machinery.
- **CLI startup loads the world.** `loadEcosystemProcedures` dynamically imports every manifest package's `dist/register.js` from its checkout, *including* bundle-dev/bundle-mcp, which re-import full duplicate dependency trees from their own node_modules (dedupe only via registry `has()` checks) — the acknowledged ~4s init (`cli.ts:289-290`).
- **Dead code / stale artifacts:** scaffold entirely; compiled `.js/.d.ts` committed inside `scaffold/src/schema/`; empty `scaffold/src/schematics/feature-add/`; `cue/git/ignore/env.cue` and `vscode.cue` unreachable (no such features); `client-lib/src/git/ref-parser.ts` mostly unused (only `isMark1Russell7Ref`); `ecosystem/src`; cli's committed `dist/`; legacy single-lockfile code paths in `cli/src/lockfile.ts`.
- **Template drift:** generated packages (via cue) lack `register.ts`, `client.procedures`, `postinstall: client announce`, `prepare`, vitest config/dep/script, and a real name — i.e., `lib new` output does not match what any actual `client-*` package looks like; every existing package was hand-upgraded after scaffolding. scaffold's `REPO_SCHEMA` likewise lacks `register.ts`/`procedures/` for procedure packages.
- **Validator inconsistency:** `lib.audit` treats missing `dist/` as an error and `--fix` creates empty `dist/` dirs (`audit.ts:168-184`), while `cue-config validate-structure` deliberately downgrades `dist` to a warning (`cue/src/cli.ts:582-589`).

## Doc drift

- **CLAUDE.md bundle pattern is wrong:** it shows `import "@mark1russell7/client-cli/register.js"`, but bundles import package roots (`bundle-dev/src/register.ts`), and `./register.js` isn't even in any client-* `exports` map — the documented import would throw `ERR_PACKAGE_PATH_NOT_EXPORTED`.
- **cli/README.md** (753 lines): documents `mark lib list` (no such procedure; registered set is scan/refresh/rename/install/new/audit/pull), `mark lib new my-package --description "…"` (no `description` field in `LibNewInputSchema`), `mark lib new my-pkg 1.0.0 -g` (no positional version, no `-g` short), and `-f, --format` (never honored — bug 4).
- **client-lib/README.md**: architecture diagrams claim DAG execution goes through `client-dag` ("Traverse --> DAG[client-dag]"); code uses the internal copy.
- **cue/README.md**: documents the `vitest` feature as adding "Vitest testing framework" (adds nothing — bug 20); feature table/graph otherwise accurate.
- **scaffold/README.md** (1,292 lines): presents scaffold as the ecosystem's scaffolding system ("No Manual File Creation", schematics quick-start) — it is unused, its cueGenerator design (`--inject feature_…`) doesn't match the cue package, and its dependencies.json template is invalid.
- **ecosystem/README.md**: accurately documents an API (`manifest`, `getPackagePath`, …) that has zero consumers; "source of truth" is true only of the JSON file, which everyone reads directly.
- **Deprecation JSDoc in client-lib** (`new.ts:8-19`, `refresh.ts:12-23`, `install.ts`, `pull.ts`): tells users to switch to `agg.*` procedures that cannot currently execute (bug 2).
- **`ecosystem.procedures` output** misattributes namespaces (claims client-cli owns `dag`/`procedure`), contradicting `register.ts`'s own "canonical home" comments.

## Roadmap candidates

1. **Fix `lib new` output end-to-end (highest value):** inject `name`/`description` into CUE (e.g. `cue eval -t name=@mark1russell7/foo` with a tagged field in base.cue, or post-write the name in lib.new); add `vitest.cue` (devDep + `test` script), a `register.ts`/`client.procedures` option for client packages, and `postinstall`. Add an e2e assertion that the generated package.json name is correct (current e2e only checks file existence).
2. **Fix or delete the aggregation layer:** either register `dag.execute`, `lib.refresh.single`, `lib.install.single`, `lib.install.cloneMissing`, `lib.pull.single`, `lib.manifest.add` (or repoint refs to the `agg.*` paths), and rewrite parity tests to run the *real* executor via `procedure.define`; or drop the agg files and the misleading `@deprecated` notices. Also resolve the duplicate `gitInitWorkflowAggregation`.
3. **Repair CUE correctness:** make `onlyBuiltDependencies` auto-population actually emit (compute from the merged feature contributions, not self-referential `output`); change the generate merge to deep-merge/preserve `exports` and `pnpm`; align `node-cjs`/`cjs.json` naming with what it emits; regenerate `cli/dependencies.json` or its tsconfig to agree.
4. **De-duplicate the DAG:** make client-lib import `client-dag` (it's generic-typed and diff-identical) and delete `client-lib/src/dag/`; remove the no-op client-dag import from bundle-dev or give client-dag real procedures (`dag.execute` from item 2 is the natural home).
5. **Make `@mark1russell7/ecosystem` real:** move manifest loading/`~` expansion there, consume it from cli/client-lib/client-cue/cue, and wire `cue-config validate-manifest` + an "unlisted packages on disk" check (would have caught `otel-ts`) into `lib.audit`.
6. **Retire scaffold** (archive the repo, or extract only the type-enforced schema-registry idea into client-lib); delete the committed build artifacts in `src/schema/` and the empty `feature-add/`.
7. **CLI polish:** reserve only `--format` (stop deleting `f`), honor procedure shorts, surface client-mode remote errors instead of silently re-running locally, and fix branch pinning (parse dependents' `github:…#branch` refs for `requiredBranch`, and gate `ensureBranch` behind `!dryRun`).
8. **Windows hygiene sweep:** `pathToFileURL` in `ecosystem/procedures.ts`, `os.homedir()` in `rename.ts`, replace `rm -rf` clean scripts in generated package.json (`ts.cue`) with a Node-based clean, fix `setupCueMod` fallback, and remove the reliance on globally-linked `cue-config`/`mark` (`npx cue-config` in lib.new only works because `@mark1russell7/cue` is npm-linked globally on this machine).
9. **Startup performance:** skip `bundle-*` packages during CLI ecosystem discovery (their contents are already loaded from source checkouts) or lazy-load registers per namespace on first use.
10. **Commit/build hygiene:** rebuild + commit the WIP zodAdapter refactors in client-lib/client-cue (dist currently stale vs src), decide on committing `dist/` consistently (cli commits it, others don't), and remove `package-lock.json` from cue/ecosystem.
