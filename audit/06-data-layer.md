# Data Layer Audit (client-mongo, client-sqlite, MiniMongo, client-fs, client-s3, client-snapshot, docker/mongo, docker/sqlite)

## Package summaries

**client-mongo** (`C:\Users\markt\git\client-mongo`, ~2,100 LOC, v1.0.0) — Wraps the `mongodb` v7 driver as 16 `mongo.*` procedures (database.ping/info, collections.list/create/drop/stats, documents.find/get/insert/update/delete/count/aggregate, indexes.list/create/drop). Key files: `src/connection.ts` (module-global singleton connection; procedures call `getDb()`, per-call `meta.database` override via `db.client.db()` — valid on driver v7 which exposes `Db.client`), `src/register.ts` (auto-registers on import), `src/storage/mongo-storage.ts` (implements `CollectionStorage<T>` from client-collections as `{_id: string, data: T}` KV docs), `src/procedures.types.ts` (hand-written typed-caller interface). Connection lifecycle is sound (one pooled MongoClient, not per-call), but all input/output schemas are pass-through (`outputSchema` = no validation). Typecheck: clean (0 errors, verified with `--incremental false`). Tests: none. Dist committed but ~15h older than working-tree src (uncommitted schema-dedup refactor in `src/procedures/schema.ts`). **Grade: B-**

**client-sqlite** (`C:\Users\markt\git\client-sqlite`, ~490 LOC) — 4 procedures: `db.query`, `db.execute`, `logs.store`, `logs.query`, all delegating to `@mark1russell7/docker-sqlite` (sql.js). Default DB `~/logs/cli/cli.db`. Data DOES survive process exit — but only via a load-entire-file / mutate-in-memory / rewrite-entire-file cycle per call (see Bugs #10–13). Prepared statements used for values, but `LIMIT`/`OFFSET` are string-interpolated. Pass-through schemas — no runtime validation. Typecheck clean; no tests. **Grade: C**

**MiniMongo** (`C:\Users\markt\git\MiniMongo`, ~8,700 LOC TS/TSX, 70 files — not 30 files/3.5k as briefed) — **Not an in-memory mongo-like store.** It is a React 19 + Vite + React Flow (XYFlow) visual MongoDB explorer: recursive type-driven renderer (`src/renderer/*`, ~6k LOC of viewers/layouts/registries), a universal RPC client provider (`src/lib/api/provider.tsx`, HTTP/WS transports + timeout/cache/retry/pagination middleware), a zustand flow store, a mock data generator, and a bootstrap script (`scripts/dev.ts`) that starts docker-mongo, a procedure server (client-server-mongo) on 0.0.0.0:5000/5001, then Vite. There are **no query/update operators, no indexes, no store engine** anywhere in it — the query-operator audit items are inapplicable. `docs/` contains 60+ stale session/status markdown files. Typecheck clean; no tests. **Grade: C+**

**client-fs** (`C:\Users\markt\git\client-fs`, ~880 LOC) — 11 `fs.*` procedures (read, write, exists, mkdir, rm, readdir, stat, copy, move, glob, read.json), all async `node:fs/promises`, each with a real Zod input schema via `zodAdapter` (the only data-layer package with genuine input validation). Windows paths handled implicitly by Node (`join` used in readdir; no explicit normalization — fine). No path allowlist/sandboxing of any kind: any registered server exposes arbitrary read/write of the whole filesystem. Error mapping is raw Node errors except where swallowed (Bugs #19–21). Typecheck clean; no tests. **Grade: B-**

**client-s3** (`C:\Users\markt\git\client-s3`, ~1,070 LOC) — 8 `s3.*` procedures (upload, download, list, delete, multipart init/upload/complete/abort) over `@aws-sdk/client-s3`, Zod-validated inputs, lazy singleton client, MinIO/LocalStack endpoint support (`forcePathStyle`). Pagination exposed correctly via `continuationToken`/`isTruncated` on `s3.list`. No streaming — everything is base64 strings through the RPC layer. Credentials strictly from `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` env vars (throws otherwise). Typecheck clean; no tests. **Grade: B**

**client-snapshot** (`C:\Users\markt\git\client-snapshot`, ~1,230 LOC) — 5 `snapshot.*` procedures meant to tar up git repos (light/medium/heavy presets), upload archive + metadata JSON to S3 (multipart >5MB, sequential parts with abort-on-fail), list/diff/restore/delete. Composes other packages via `ctx.client.call` (s3.\*, git.\*). In reality **snapshot.create always crashes** (`require()` in ESM) and would fail anyway (tar cwd bug), and **restore without `targetPath` deletes what it just restored**. Checksums are computed but never verified. Typecheck clean; no tests. **Grade: F**

**docker/mongo** (`@mark1russell7/docker-mongo`, ~370 LOC) — docker-compose.yml (mongo:8.0, healthcheck, volumes, env-configurable port/name, **no authentication configured**) plus TS utilities: `withConnection` (new MongoClient per call, always closed — correct for its seeding/init use-case), `initDatabase` (collections + indexes, idempotent), `seedDatabase` (`clearFirst` defaults to **true** — destructive by default). Typecheck clean; no tests. **Grade: B**

**docker/sqlite** (`@mark1russell7/docker-sqlite`, ~530 LOC) — Despite the name, contains no Docker-dependent code: sql.js (WASM SQLite) helpers — `withConnection` (load file → run → export → rewrite file), query/execute/execMultiple/lastInsertRowId/tableExists, and a versioned migration system (`_migrations` table, up/down, rollback). The compose file just runs `coleifer/sqlite-web` UI on 8080 (no auth). Durability model is the package's core weakness (Bug #12). Typecheck clean; no tests. **Grade: C+**

## Bugs

1. **snapshot.create crashes: `require()` in an ESM package** — `C:\Users\markt\git\client-snapshot\src\procedures\snapshot\create.ts:63` (also :71, :76, :233): `require("fs").writeFileSync(metadataPath, JSON.stringify(metadata, null, 2));` Package is `"type": "module"` and compiled dist retains `require(` (verified in `dist/procedures/snapshot/create.js:42`); `require` is undefined in ES-module scope, so every `snapshot.create` call throws `ReferenceError` before archiving. **Severity: critical. Confidence: certain.**

2. **snapshot.create tars from the wrong directory** — `create.ts:273-286`: `await tar.create({ ..., cwd: workDir, filter: ... }, repoPaths.map((p) => basename(p)))`. `workDir` is a fresh temp dir (`$TMP/snapshot-<id>`); the repos are not inside it, so tar entries like `client-mongo` don't exist under `cwd` → ENOENT (and even if tolerated, the archive would never contain the repos). Feature is doubly dead with #1. The `files` array built at :246-256 is dead code. **Severity: critical. Confidence: certain.**

3. **snapshot.restore without `targetPath` deletes everything it restored** — `restore.ts:82` sets `const targetPath = input.targetPath || workDir;` then the `finally` at :124-128 runs `if (existsSync(workDir) && workDir !== input.targetPath) rmSync(workDir, {recursive: true, force: true})`. When `input.targetPath` is undefined, extraction goes into `workDir`, and `workDir !== undefined` is true → the just-restored files are wiped, yet the procedure returns `success: true` with `restoredPaths` pointing at deleted paths. **Severity: high. Confidence: certain.**

4. **Snapshot lookup is unpaginated and substring-matched** — `restore.ts:34-45`, `diff.ts:26-36`, `delete.ts:19-31`: one `s3.list` call (S3 caps at 1000 keys, no continuation loop) then `obj.key.includes(input.id)`. With >1000 objects under `snapshots/`, snapshots silently become "not found"; and `includes` can match a different snapshot whose id contains the given id as a substring — for `snapshot.delete` that deletes the wrong files. **Severity: medium (high for delete). Confidence: certain (logic).**

5. **restore overwrite-protection checks the wrong paths** — `restore.ts:89-97` and :110-115 build `join(targetPath, repo.name)` where `repo.name` is the package.json name (e.g. `@mark1russell7/client-mongo`), but archive entries are directory basenames (e.g. `client-mongo`). Scoped names produce paths like `<target>\@mark1russell7\client-mongo` that never exist → overwrite check is a no-op and `restoredPaths` is wrong. **Severity: medium. Confidence: certain.**

6. **Exclude patterns don't support globs** — `create.ts:278-283`: `if (path.includes(pattern))` with pattern `"*.log"` — literal substring never matches any path; log files are always included. Also `"node_modules"` substring matching excludes any path merely containing that string. **Severity: low. Confidence: certain.**

7. **Restore never verifies the stored checksum** — `metadata.checksum` (SHA-256, computed at `create.ts:70`) is never recomputed/compared in `restore.ts` despite README's "checksums for verification". **Severity: medium. Confidence: certain.**

8. **Command injection via snapshot metadata** — `diff.ts:109-111`: ``execSync(`git diff --stat ${snapshotRepo.commit}..HEAD`, ...)`` where `commit` comes from JSON downloaded from S3. Anyone able to write the bucket can execute shell commands on the machine running `snapshot.diff`. **Severity: medium (security). Confidence: certain (code path), exploitability possible.**

9. **client-mongo has zero runtime input validation** — `C:\Users\markt\git\client-mongo\src\procedures\schema.ts:1` re-exports the core's `outputSchema` (pass-through; `client/src/procedures/core/schemas.ts:99-105` returns data unvalidated) and every procedure uses it for inputs. Malformed input reaches the driver raw: e.g. `documents.find.ts:51-53` — `page: 0` → `skip = -20` → `MongoInvalidArgumentError`; string `limit` → NaN. Same pattern in client-sqlite. **Severity: medium. Confidence: certain.**

10. **SQL injection / syntax breakage via interpolated LIMIT/OFFSET** — `C:\Users\markt\git\client-sqlite\src\register.ts:210-211`: `` const limitClause = input.limit !== undefined ? `LIMIT ${input.limit}` : ""; `` (same for OFFSET). Input is unvalidated (pass-through schema), so a string `limit` is spliced directly into SQL executed against the DB. All other values are parameterized; these two are not. **Severity: medium. Confidence: certain.**

11. **logs.query with `offset` but no `limit` generates invalid SQL** — `register.ts:210-219`: SQLite's grammar only allows `OFFSET` as part of a `LIMIT` clause; `ORDER BY timestamp DESC OFFSET 10` is a syntax error → procedure throws. **Severity: medium. Confidence: certain.**

12. **docker-sqlite persistence is read-modify-rewrite with no locking or atomicity** — `C:\Users\markt\git\docker\sqlite\src\connect.ts:80-94`: every `withConnection` loads the whole DB file, runs the callback, then `saveDatabase` re-exports and `writeFile`s the entire DB — even for pure SELECTs (`client-sqlite` `db.query` rewrites the file on every read). Two concurrent calls (trivially possible behind an async procedure server) both load, both write; last writer silently discards the other's committed rows. `writeFile` is also non-atomic (no temp+rename), so a crash mid-write corrupts the database. **Severity: high (for anything beyond single-shot CLI use). Confidence: certain (design); loss occurrence likely under concurrency.**

13. **logs.store does two full load/rewrite cycles per log entry** — `client-sqlite/src/register.ts:145-147`: `ensureLogsTable(dbPath)` (open→migrate→save→close) then a second `withConnection` for the INSERT (open→insert→save→close). Doubles I/O and widens the #12 race window on the hottest path. **Severity: low (perf) feeding #12. Confidence: certain.**

14. **client-mongo `connect()` leaks the previous connection** — `C:\Users\markt\git\client-mongo\src\connection.ts:86-126`: creates a new `MongoClient`, connects, and `setDefaultConnection()` unconditionally; a prior default connection is orphaned open (its pool persists). `client-server-mongo`'s `server.mongo.start` calls `connect()` on every start, so repeated starts accumulate clients. **Severity: medium. Confidence: certain.**

15. **parseId makes 24-hex string `_id`s unreachable** — `client-mongo/src/procedures/documents.get.ts:29-35` (dup in update/delete): any string that parses as an ObjectId is coerced (`new ObjectId(id)`), with no way to force a string match. Documents stored with a 24-hex *string* `_id` — exactly what `MongoStorage` (`src/storage/mongo-storage.ts:81-87`) writes if a caller uses hex-ish keys — can never be fetched/updated/deleted by id. Real Mongo tools query both alternatives (`{$or: [{_id: oid}, {_id: str}]}`) or take an explicit type. **Severity: medium. Confidence: likely.**

16. **`procedures.types.ts` typed-caller interface contradicts the implementations** — `C:\Users\markt\git\client-mongo\src\procedures.types.ts`: `UpdateInput` declares `query: DocumentQuery` (:186) but the handler reads `filter`/`id` (`documents.update.ts:17-28`); `DropCollectionInput` declares `{name}` (:100) but the handler requires metadata collection + `{confirm: true}` (`collections.drop.ts:13-45`); `CreateCollectionOutput {created,name}` vs actual `{acknowledged,collectionName}`; `CreateIndexOutput {name}` vs actual `{acknowledged,indexName}`; `DropIndexInput {name}` vs actual `{indexName}`; `PingInput.timeout` and `ListCollectionsInput.filter` are silently ignored by handlers. Anyone using `createTypedCaller<MongoProcedures>` writes calls that typecheck and fail at runtime. **Severity: medium. Confidence: certain.**

17. **fs.rm swallows all errors when `force: true`** — `C:\Users\markt\git\client-fs\src\procedures\fs\rm.ts:17-25`: `try { await stat(input); await fsRm(...); } catch (err) { if (force) return {path, removed: false}; throw err; }`. Node's `rm` `force` only ignores ENOENT, but this catch converts EPERM/ENOTEMPTY/EBUSY into a quiet `{removed: false}`. The pre-`stat` is also TOCTOU and redundant. **Severity: medium. Confidence: certain.**

18. **fs.mkdir reports success when the path is an existing file** — `mkdir.ts:15-22`: `stat` succeeding (file OR dir) returns `{created: false}` as if the directory exists; and any non-ENOENT stat failure falls into the mkdir attempt. **Severity: low. Confidence: certain.**

19. **fs.move cross-device fallback can drop data on races / wrong force semantics** — `move.ts:30-38`: on EXDEV it calls `copy({src, dest, recursive: true, overwrite})`; with `overwrite: false`, `cp`'s `force: false` **silently skips** existing dest files (no `errorOnExist`), after which `rm({path: src, recursive: true, force: false})` deletes the source — content that was skipped is lost. Protected only by the earlier non-atomic exists-check (TOCTOU). **Severity: medium. Confidence: possible (needs race or pre-existing dest created between check and copy).**

20. **fs.glob depends on Node ≥22 API but package declares no engines** — `glob.ts:7`: `import { glob as nodeGlob } from "node:fs/promises"` — added in Node v22 (experimental). `client-fs/package.json` has no `engines`; sibling packages state `node >=20`; on Node 20 `nodeGlob` is `undefined` → TypeError at call time. README claims "fast-glob" is used (it isn't, and isn't a dependency). **Severity: medium. Confidence: certain (code/docs), runtime impact depends on Node version.**

21. **fs.read returns a Buffer when invoked without schema-applied defaults** — `read.ts:14-16` casts `encoding as BufferEncoding`; the utf8 default lives only in the Zod schema (`types.ts:19`). Call paths that invoke `procedure.handler` directly without `safeParse` — e.g. `MiniMongo/scripts/dev.ts:100-116` transport registration — pass `encoding: undefined`, making `readFile` return a Buffer that then JSON-serializes as `{type:"Buffer",data:[...]}` in a field typed `string`. **Severity: low. Confidence: likely.**

22. **fs.readdir recursive: no symlink-cycle guard; one bad entry aborts the listing** — `readdir.ts:14-38`: recurses into directories via `withFileTypes` (symlinked dirs → potential infinite loop on cyclic links) and `includeStats` stats every entry with no try/catch (a broken symlink rejects the whole call). **Severity: low. Confidence: likely.**

23. **client-s3 rejects the AWS default credential chain and session tokens** — `C:\Users\markt\git\client-s3\src\s3-client.ts:16-41`: hard-requires `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` (throws otherwise) and never reads `AWS_SESSION_TOKEN` — so SSO/STS temporary credentials (which require the token) authenticate incorrectly, and IAM-role/profile setups (explicitly recommended by its own README line ~819) cannot work at all. Simply omitting `credentials` would let the SDK resolve the full chain. **Severity: medium. Confidence: certain.**

24. **s3.download buffers whole objects into memory as base64** — `download.ts:40-56`: `transformToByteArray()` then base64 string (+33% size) through the RPC envelope; no size cap or streaming. Combined with snapshot restore (which downloads multi-GB archives this way, `restore.ts:67-76`), large objects will exhaust memory. **Severity: medium. Confidence: certain.**

25. **MiniMongo service layer expects a response shape the server never sends** — `C:\Users\markt\git\MiniMongo\src\services\api.ts:35-58`: reads `response.pagination.currentPage` / `totalDocuments`, but `mongo.documents.find` returns `pagination.page` / `total` (`client-mongo/src/procedures/documents.find.ts:69-78`) → mapped values are `undefined` (currently latent: `CollectionListViewer.tsx:119-123` only uses `.documents`). `getDocument` (:61-71) returns the raw response typed as `MongoDocument`, but the procedure returns `{document: ...}` (`documents.get.ts:59-61`) → wrong shape for any future consumer. **Severity: medium (latent). Confidence: certain (cross-verified).**

26. **Two incompatible path→Method mappings in the ecosystem** — `MiniMongo/scripts/dev.ts:66-70` maps `["a","b","c"]` → `{service:"a", operation:"b.c"}`, while core `Client`/`ProcedureServer` map to `{service:"a.b", operation:"c"}` (`client/src/client/client.ts:954-965`, `client/src/server/procedure-server.ts:361-372`). dev.ts is internally self-consistent today, but any cross-wiring (e.g. pointing the UI at the dev-script transport) fails method resolution silently. **Severity: low. Confidence: certain.**

27. **MiniMongo remote-bridge misuses React `use()`** — `src/renderer/bridge/index.tsx:259-263`: `renderer(ctx)` is invoked during render, producing a fresh Promise each render, then `use(result)` — unstable promise → perpetual re-suspension when the remote path is enabled (currently dormant: nothing calls `configureBridge` with a `call`). Also `DynamicViewer.tsx:118-125` renders resolved children with `EMPTY_CONTEXT`, dropping `ctx.api` for nested dynamic values. **Severity: low (dormant). Confidence: likely.**

28. **Unauthenticated network exposure of the full procedure registry** — `MiniMongo/scripts/dev.ts:255-268` starts the peer with `host: "0.0.0.0"` (HTTP :5000, WS :5001) after importing client-shell/client-docker; `server.create` with `autoRegister` (`client-server/src/procedures/server.create.ts:33-44`) exposes **every** registered procedure — including `shell.run`/`shell.exec` and all `fs.*` — with no authentication (peer has none; default host is also `0.0.0.0` at `client-server/src/peer/index.ts:135`). 29. **docker-mongo compose runs MongoDB 8 with no auth on a published port** — `C:\Users\markt\git\docker\mongo\docker-compose.yml`: no `MONGO_INITDB_ROOT_USERNAME/PASSWORD`, port `${MONGO_PORT:-27017}:27017` published (Docker publishes on 0.0.0.0 by default). ## Architecture flaws

- **The name-vs-reality problem**: MiniMongo is a UI, not a store; `docker-sqlite` contains no Docker-coupled code (sql.js + an optional web-UI compose); `docker-mongo` is 90% generic TS utilities. The briefed mental model ("MiniMongo = in-memory mongo-like store") does not survive contact with the code — a naming/layering smell in itself.
- **Three disjoint storage abstractions with no shared contract**: (1) `CollectionStorage<T>` (client-collections) implemented by `MongoStorage` as `{_id, data}` KV; (2) the `mongo.*` document-procedure surface; (3) the `db.*`/`logs.*` SQLite procedure surface. A caller cannot swap Mongo↔SQLite behind a common interface; `MongoStorage` documents written via (1) are second-class citizens to (2) (wrapped in `data`, hex-string ids unreachable per Bug #15). There is no `sqlite` implementation of `CollectionStorage`, and no `mongo.*`-equivalent generic surface for SQLite.
- **Inconsistent connection management per package**: client-mongo = process-global mutable singleton set only by programmatic `connect()` (no `mongo.connect` procedure — a bare procedure server hosting client-mongo throws "connection not initialized" unless a bootstrap like `server.mongo.start` runs); docker-mongo = new client per call; docker-sqlite = full file load/rewrite per call; client-s3 = lazy env-frozen singleton. Four packages, four lifecycles, no shared pattern or pooling policy.
- **Validation is split-brain**: client-fs/client-s3/client-snapshot use real Zod; client-mongo/client-sqlite (and client-server) use pass-through `outputSchema` for inputs. The ecosystem's premise ("procedures are path-addressed commands with Zod schemas") holds for only half the data layer, and whether even real schemas run depends on the call path (ProcedureServer validates; MiniMongo's dev.ts transport registration calls handlers raw).
- **`sideEffects: false` on packages whose register.js works by import side effect** (client-mongo/fs/s3/sqlite package.json) — any bundler consuming `register.js` may tree-shake the registration away. Currently unexercised (registration happens in Node servers), but it's a landmine.
- **Duplicated helper code drifting**: `zodAdapter`/`ZodLikeSchema` were copy-pasted into client-fs/client-s3/client-snapshot registers and are mid-migration to the core (uncommitted edits in all four repos; dist ~15h stale vs src). `parseId`/`buildFilter` duplicated across three client-mongo procedures. dev.ts duplicates the registry→transport sync "from server/src/cli.ts" with a divergent method mapping (Bug #26).
- **RPC-shaped I/O forces base64 through JSON for binary** (s3, snapshot): no streaming anywhere in the layer; snapshot round-trips multi-GB archives through in-memory base64 strings twice.
- **No tests anywhere in scope** (zero `*.test.ts`/`*.spec.ts`/vitest configs across all 8 packages), which is how a fully-broken package (client-snapshot) can typecheck clean and sit in main. Committed `dist/` as the distribution mechanism (git-install) plus no CI means stale-build skew is a standing risk.

## Doc drift

- **client-mongo README** documents the fictional `procedures.types.ts` API, not the implementation: `collections.drop` with `{name}` (actual: metadata collection + `{confirm: true}`); `documents.update` with `query` (actual `filter`/`id`); `collections.create` output `{created,name}` (actual `{acknowledged,collectionName}`); `indexes.create/drop` I/O shapes; `database.ping` `{timeout}` and `collections.list` `{filter}` params that handlers ignore.
- **client-fs README**: "Pattern matching: Glob file search with **fast-glob**" and an architecture diagram showing fast-glob — code uses experimental `node:fs/promises` glob; fast-glob is not a dependency. "Zod validation: Runtime type safety on all inputs" — true only on validating call paths.
- **client-s3 README**: "Use IAM roles when running on AWS infrastructure" — impossible; the client throws without static env keys (Bug #23).
- **client-snapshot README**: essentially all feature claims ("Complete State Capture", "Multipart Upload", "Restore Safety", "checksums for verification") are false in practice — create crashes (Bugs #1–2), restore self-deletes (#3), checksum unverified (#7).
- **docker-mongo README**: "Production-ready Docker Compose" (no auth, no resource limits); "Connection Pool" in the architecture diagram (a client is created and closed per `withConnection` call).
- **docker-sqlite**: package name/description "Reusable SQLite **Docker** setup" — the library is Docker-free; compose is an optional viewer. README's "Auto-save functionality" euphemizes the rewrite-whole-file model without mentioning the concurrency hazard.
- **MiniMongo README**: "Real-Time Updates: WebSocket support for live data synchronization" — WS is just an alternate request/response transport; no live sync/subscriptions exist. "MongoDB Integration ... with connection pooling" — pooling is the server's, not the app's. Also 60+ stale progress docs in `docs/` (SESSION-COMPLETE, FINAL-STATUS, etc.) that describe past states as current.
- **CLAUDE.md architecture doc** describes client-* packages as thin CLI wrappers over client-shell; the data layer doesn't fit that model (direct SDK/driver usage), which is fine, but the manifest-of-record (`dependencies.json`) is just cue feature flags, not the described "ecosystem manifest" — `snapshot.create`'s `getEcosystemPaths` is still a TODO returning `[process.cwd()]` (`create.ts:131-135`), so the documented "all repos in ecosystem" default is fiction.

## Roadmap candidates

1. **Fix or retire client-snapshot** (highest ROI: 4 certain critical/high bugs in ~300 lines): replace `require()` with `node:fs` imports; tar with `cwd: dirname(repoPath)` per repo (or `-C` style multi-cwd handling); make `targetPath` required or default to a real location and never rm it; verify checksum on restore; paginate + exact-match (`snapshots/<name>/<id>.tar.gz`) instead of `includes`; stream archives via presigned URL or file-based multipart instead of in-memory base64.
2. **Make validation real and single-sourced**: give client-mongo and client-sqlite genuine Zod input schemas (page/limit bounds, LIMIT/OFFSET as integers, confirm booleans); finish the in-flight `zodAdapter`/`outputSchema` centralization (uncommitted in 4 repos) and rebuild/commit dist; ensure every transport path (including dev.ts-style registration) runs `safeParse`.
3. **Replace sql.js persistence with `node:sqlite` (Node 22+) or better-sqlite3** behind the existing `withConnection` API — real WAL durability, real concurrency, no whole-file rewrites; short-term mitigation: temp-file+rename atomic save, in-process mutex per dbPath, and skip save for read-only ops.
4. **Security baseline for peer servers**: default `host: 127.0.0.1`; procedure allowlist/denylist for `autoRegister` (never expose `shell.*`/`fs.*` unauthenticated); optional bearer-token middleware; add auth to docker-mongo compose (`MONGO_INITDB_ROOT_USERNAME/PASSWORD` + `MONGODB_URI` with creds) and bind published ports to 127.0.0.1.
5. **client-s3 credential chain**: pass explicit credentials only when env keys exist, else let the SDK default chain resolve (adds SSO/IMDS/profile + `AWS_SESSION_TOKEN` support for free); add a `s3.listAll` helper that loops continuation tokens.
